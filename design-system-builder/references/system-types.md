# System Types

The three system types are default templates, not fixed packages. If the user or future UI provides a custom project configuration, follow the custom include/exclude list over the template.

## Shared Requirements

Every Figma write requires:

- Figma link
- Brand primary color
- Brand primary gradient stops and direction, if the brand primary is a gradient
- Background mode for shadow decisions: ask whether the main background is light or dark. If the designer does not specify, assume a light background.
- Grey mode: opacity grey or solid neutral scale
- Black or gray base color for grey tokens; required, designer-provided, and not `#000000`
- White base color for `Neutrals/White`, default `#FFFFFF` if not provided

Optional inputs:

- Brand text color
- Solid fallback color for a brand gradient
- Auxiliary base color, only if the designer wants auxiliary colors
- Auxiliary helper mode: solid white-mix or opacity, only if auxiliary colors are used
- Predefined semantic colors
- Custom radius, spacing, typography, or shadow list

Typography inputs:

- Font family is required before writing text styles.
- One font family is the default recommendation.
- One to three font families are allowed when the user provides a clear mapping.
- If two or three font families are used, ask which font is primary and where each secondary font is used. Common secondary uses include headline accent words, full display headlines, numbers, prices, brand display text, English-only text, or a specific module title.
- The primary font gets the complete typography preset. Secondary fonts only get purpose-based partial styles.
- If primary and secondary fonts are mixed in the same headline or text block, use the primary font's size and line height as the reference, then adjust the secondary font size, weight, and line height until the visual size matches.
- More than three font families is discouraged; ask for confirmation or simplification before writing.

Do not ask for app name, audience, or product category unless the user wants creative color/style suggestions.

## Lightweight Product

Use for small consumer mobile apps, MVPs, or products that need quick visual consistency without a heavy system.

Default behavior: style-first. In a future agent or UI, preselect `Style` for every lightweight module and make `Variables` an optional toggle. Small projects should not receive Figma variable collections unless the user explicitly enables variables or provides a custom configuration that includes them.

Generate as styles where Figma supports styles:

- `Neutrals/Grey` opacity paint styles
- `Neutrals/White` opacity paint styles
- Or `Neutral/900` through `Neutral/00` solid neutral paint styles if solid neutral scale is selected
- `Brand` primary paint style
- `Brand` primary gradient paint style, if provided
- `Auxiliary/base`, `Auxiliary/02`, and `Auxiliary/03` only if the designer wants auxiliary colors
- Basic `Semantic` paint styles: success, warning, error; info is optional
- Compact typography preset from `typography-presets.md`
- Optional `Shadow/S`, `Shadow/M`, and `Shadow/L` effect styles if useful

Handle modules without native Figma styles:

- Radius: document a small radius set with values in names; create number variables only if variables are enabled.
- Spacing: document a small spacing set using multiples of 4; create number variables only if variables are enabled.

If variables are enabled, create the lightweight color and number variables using the same naming rules in `token-rules.md`.

Avoid:

- Default variable collections
- Multi-theme modes
- Complex alias layers
- Full component libraries
- Frontend token export unless explicitly requested

## Standard Product

Use as the default recommendation for a complete consumer mobile app design system.

Required inputs:

- Figma link
- Brand primary color, or brand primary gradient plus a solid fallback
- Background mode for shadow decisions; default to light if not specified
- Grey mode: opacity grey or solid neutral scale
- Black or gray base color; required, designer-provided, and not `#000000`
- White base color, default `#FFFFFF`

Generate:

- `Brand`
  - primary
  - primary gradient paint style only if provided
  - two lighter colors derived from primary
  - two darker colors derived from primary
  - brand text color, inferred if not provided
- `Neutrals/Grey`
  - opacity variables based on the black/gray base, if opacity grey is selected
- `Neutral`
  - `Neutral/900` through `Neutral/00` solid variables, if solid neutral scale is selected
  - `Neutral/900` uses the designer-provided black/base color
  - `Neutral/00` uses `#FFFFFF`
  - intermediate values are generated by perceptual lightness in OKLCH/OKLab/LAB-like space
- `Neutrals/White`
  - opacity variables based on white, if opacity grey is selected
- `Semantic`
  - complete success, warning, error, info colors
  - 10% background/helper color for each semantic color
  - familiar hue families preserved: success green, warning yellow/orange, error red, info blue/cyan
  - generated colors adapted to the brand primary's lightness, saturation, and mutedness/greyness
- `Auxiliary`
  - optional; ask whether auxiliary colors are needed if none are provided
  - `Auxiliary/base` from one designer-provided key color
  - `Auxiliary/02` from 50% white-mix or 50% opacity, depending on designer preference
  - `Auxiliary/03` from 90% white-mix or 10% opacity, depending on designer preference
- `Radius`
  - named with concrete pixel values
  - default set includes `4px`, `8px`, `12px`, `16px`, `20px`, `24px`, `28px`, `32px`, and `999px`
  - capsule radius `999px`
- `Spacing`
  - all multiples of 4
  - default set includes `4px`, `8px`, `12px`, `16px`, `20px`, `24px`, `28px`, `32px`, `40px`, and `48px`
- Text styles
  - compact or full typography preset from `typography-presets.md`; ask which preset to use if the user has not specified
- `Shadow/S`, `Shadow/M`, and `Shadow/L` effect styles

For standard product systems, solid colors must be variables only. Do not create duplicate solid-color paint styles. Use paint styles only for gradients or other fills that Figma color variables cannot represent.

Do not generate components until variables and styles are complete and the user confirms components are needed.

## Multi-Theme

Use when the project needs Light/Dark, multiple brands, seasonal skins, premium/default themes, or any other theme switching.

First determine the multi-theme subtype:

- Light/Dark subtype: the designer selected multi-theme and provided only one brand primary color. Default to Light and Dark modes.
- Multi-brand / multi-primary subtype: the designer provided two or more brand primary colors.

### Light/Dark Subtype

Use this subtype when the designer provides one primary color for a multi-theme system.

Required inputs:

- Figma link
- One brand primary color, or one brand primary gradient plus a solid fallback
- Grey mode: opacity grey or solid neutral scale
- Black or gray base color; required, designer-provided, and not `#000000`
- White base color, default `#FFFFFF`

Generate Light and Dark modes:

- Light mode may use the provided primary color directly or lightly adapt it.
- Dark mode should derive a dark-suitable primary by slightly increasing perceived lightness and reducing saturation/chroma so it does not feel neon, harsh, or muddy on dark backgrounds.
- Rebuild dark-mode backgrounds, surfaces, text colors, borders, semantic colors, and shadows for dark context. Do not invert the light palette mechanically.
- Avoid common dark-mode issues: large pure-black areas, pure-white body text, high-saturation glowing brand colors, low-contrast borders, heavy shadows copied from light mode, and semantic colors copied directly from light mode.
- Dark-mode semantic colors should preserve familiar hue families but reduce saturation and tune lightness for dark backgrounds.
- Dark-mode shadows should use the lightest derived primary color at low opacity, stay subtle, and rely more on surface contrast, borders, or restrained primary-tinted shadows than large black drop shadows.

### Multi-Brand / Multi-Primary Subtype

Use this subtype when the designer provides two or more primary colors.

Required inputs:

- Figma link
- Theme count and theme names
- Brand primary color for each theme, or brand primary gradient plus a solid fallback for each theme
- Shared background mode by default: all themes use the same light or dark background system unless the designer provides per-theme backgrounds
- One hue-neutral grey base for shared grey opacity tokens; required, designer-provided, and not `#000000`
- White base color, default `#FFFFFF`

Background handling:

- The background system may be shared across all primary themes.
- The background system may be different per theme only when the designer explicitly asks for per-theme backgrounds or provides per-theme background colors.
- Do not assume each primary theme needs a different background color.
- If background mode is not specified for a multi-brand / multi-primary system, assume all themes use a light background for shadow decisions.

Grey handling:

- Use one shared hue-neutral grey system by default.
- Prefer shared opacity grey tokens for multi-brand / multi-primary systems so the grey system works across all theme colors.
- Avoid per-theme tinted greys by default; they make multi-brand systems harder to maintain.

For each theme, generate:

- Brand primary
- brand primary gradient paint style only if provided
- two lighter primary shades
- two darker primary shades
- complete semantic colors and 10% helper backgrounds
- semantic colors preserve familiar hue families and adapt independently to that theme's brand primary lightness, saturation, and mutedness/greyness
- optional `Auxiliary/base`, `Auxiliary/02`, and `Auxiliary/03` if the designer wants auxiliary colors; adapt per theme when auxiliary base colors differ by theme

Shared across themes by default:

- Radius
- Spacing
- Full typography preset from `typography-presets.md` unless the user provides a custom type scale
- Shared hue-neutral grey opacity tokens for multi-brand / multi-primary systems, unless the designer explicitly asks for per-theme greys
- `Shadow/S`, `Shadow/M`, and `Shadow/L` effect styles; use each theme's darkest derived primary for light backgrounds and lightest derived primary for dark backgrounds

For multi-theme systems, solid colors must be variables only. Do not create duplicate solid-color paint styles. Use paint styles only for gradients or other fills that Figma color variables cannot represent.

Use Figma variable modes for themes. Example modes: `Light`, `Dark`, `Premium`, `Seasonal`, or the names provided by the user.
